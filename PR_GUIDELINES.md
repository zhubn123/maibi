# PR 提交规范

## 基本原则

1. 所有新功能必须通过 Pull Request 提交。
2. 每个 PR 只做一件事：只实现或修改单一功能。
3. 鼓励 PR 尽可能小、粒度尽可能细。
4. 大功能必须拆分为多个可独立评审、可独立运行的 PR 分步提交。
5. 每个 PR 合并后，主分支代码必须保持可运行状态，评委在任意时间查看都应能复现演示效果。

## PR 标题与描述

PR 标题必须用一句话说明本 PR 新增或修改了什么。

PR 描述必须包含以下内容：

- 功能描述：说明该功能的作用与使用方式。
- 实现思路：简要说明技术选型或核心实现逻辑。
- 测试方式：说明如何验证该功能正常运行。
- 依赖说明：如引用第三方库或框架，必须列明依赖，并同步写入 README。
- 原创性说明：如复用了自己过去的代码片段，必须注明来源和改动范围。

## PR 描述模板

```markdown
## 功能描述

说明本 PR 实现了什么功能，用户如何使用。

## 实现思路

说明核心实现逻辑、技术选型、关键模块或数据流。

## 测试方式

列出已执行的验证方式，例如：

- [ ] 本地启动客户端成功
- [ ] 本地启动服务端成功
- [ ] 单元测试通过
- [ ] 手工验证核心流程

## 依赖说明

列出新增第三方依赖、版本、用途，并确认 README 已更新。

## 原创性说明

说明是否复用历史代码；如有复用，注明来源、改造内容和原创新增部分。
```

## 不符合规范的情况

以下情况不得合并：

- PR 描述为空。
- PR 描述与实际代码变更严重不符。
- 一个 PR 同时包含多个无关功能或大范围无关重构。
- 引用了第三方库或框架，但没有在 README 中列明依赖。
- 引用了第三方库或框架，但没有说明本项目原创功能部分。
- 复用了自己过去的代码片段，但没有在 PR 描述中注明来源。
- 合并后主分支无法启动、无法运行或无法复现演示效果。

## 仓库目录约定

如果项目包含多个独立模块，应提交至同一仓库的不同子目录分别管理，便于部署和启动。

麦笔项目默认目录约定：

- `client/`：Windows 桌面客户端。
- `server/`：签名服务和后端 API。
- `core/`：客户端与服务端共享的核心类型、接口和工具。
- `docs/`：产品、设计、部署和评审文档。

## 推荐 PR 拆分顺序

1. 项目骨架与 README。
2. 共享核心接口和配置类型。
3. 签名服务最小 API。
4. Mock ASR 集成测试。
5. 客户端托盘和浮窗基础 UI。
6. 麦克风采集与音频分片。
7. 腾讯云 ASR Provider。
8. 文本上屏能力。
9. 热词、用量上限和隐私日志检查。
10. 打包、演示脚本和最终验收文档。
